# "🧩 구조화 사실(그래프)" 상세 — 무엇이고 어떤 코드 경로로 만들어지나 (2026-06-30)

화면의 "🧩 구조화 사실(그래프)" 영역이 무엇이고, 어떤 코드 경로로 만들어져 화면에 뜨는지 처음부터 끝까지 설명한다.

## 1. 큰 그림 — 두 개의 검색 레이어

같은 운영기준 문서를 **두 가지 형태로** 저장한다.

- **청크(:Chunk)** — 문서를 문단·표 단위로 잘라 **원문 텍스트 그대로** 보존. 검색의 주 대상(벡터 임베딩 + 풀텍스트).
  장점은 빠짐없이 들어있다는 것, 단점은 PDF 표가 1차원으로 뭉개지는 등 **형태가 지저분**할 수 있다는 것.
- **구조화 사실(:Entity, origin='structured')** — 같은 내용을 LLM이 미리 읽어 **"엔티티 — 속성(값) — 관계"의 정제된 단위**로
  뽑아 그래프에 넣어둔 것. 청크가 "문단"이라면 이건 "사실 카드".

화면의 "🧩 구조화 사실" 영역은 두 번째 레이어에서 **이번 질문과 관련된 사실 카드만 골라** 보여주는 곳이다.

## 2. 데이터 모델 — 사실 카드의 구조

스키마를 **TBox(온톨로지·고정 클래스) / ABox(인스턴스·실제 값)** 로 나눠 보면 다음과 같다.

![GraphRAG 스키마 — TBox / ABox](graphrag_TBox_ABox.svg)

```
(:Entity { id, label, etype, origin:'structured', source_file, …값속성 })
(:Entity)-[:REL { relation, source_file, origin:'structured' }]->(:Entity)
```

- **etype(유형)** — 화면의 녹색 배지. 고정 화이트리스트 7종: `Product` · `RateGrade` · `Nego` · `Decision` ·
  `VehicleCondition` · `HandlingRule` · `Term`. 유형을 못 박아 **스키마 드리프트**(LLM이 매번 새 유형 생성)를 방지.
- **label(이름)** — 굵은 글씨. 사실 카드의 제목(예: "차량가격 취급기준").
- **값 속성(properties)** — `rule: …` 또는 칩. 핵심 가치. 예) `RateGrade {grade:2, gl_rate:"21.0%", max_months:72}`,
  `Decision {code:"R", result:"필터링 취급 불가"}`.
- **origin='structured'** — 청크/기존 개념 그래프와 구분하는 태그. 재적재 시 이 태그만 지우고 다시 넣어 **멱등성** 보장.

## 3. 만들어지는 과정 — `build_structured`

`backend/app/scripts/build_structured.py` 동작 순서:

1. source 문서를 청크로 파싱(검색용 청크와 동일).
2. 각 청크 텍스트를 **선택 LLM(Qwen/HCX)** 에 주고, `ingestion/structured.py`의 프롬프트로 "엔티티·관계·값을
   **아래 JSON 형식으로만** 추출"하라고 지시. 프롬프트에 허용 etype 7종 + 속성 예시를 명시하고,
   **표가 깨져 있어도 행·열을 최대한 복원해 값을 채우되 불확실하면 생략**하라고 함.
3. JSON 파싱 후 **화이트리스트 검증** — etype/relation이 허용 목록이 아니면 폐기(`BadType`/`BAD_REL` 제거).
4. 이름 정규화로 **결정적 id**(`etype::해시`) 생성 → 여러 청크의 같은 엔티티를 **하나로 병합(dedup)**.
5. Neo4j 적재. 재실행 시 `origin='structured'`인 것만 지우고 다시 넣음(멱등).

이 영역의 품질은 **이 추출 단계 결과물**에 전적으로 달려 있다.

## 4. 질문이 들어왔을 때 — 검색·답변 흐름

`cypher_qa.py` 처리 순서:

1. Hybrid로 **청크** 검색(벡터+풀텍스트→RRF 상위 6).
2. `_structured_facts(question, kws)`가 **구조화 사실**을 점수순으로 조회(개선 ①② 반영):
   - **라벨 + 값 속성 + 이웃(관계 대상) 라벨**까지 매칭하고 `label_hits*2 + prop_hits*2 + nbr_hits`로 점수화.
   - `_intent_etypes(question)`로 질문 의도에 맞는 etype이면 **+5점** 가산 → `ORDER BY score DESC LIMIT 15`.
   ```cypher
   MATCH (e:Entity) WHERE e.origin='structured'
   OPTIONAL MATCH (e)-[r:REL]->(n:Entity)
   WITH e, collect(DISTINCT {rel:r.relation, target:n.label}) AS rels0
   WITH e, [x IN rels0 WHERE x.target IS NOT NULL] AS rels,
     size([kw IN $kws WHERE toLower(e.label) CONTAINS toLower(kw)]) AS label_hits,
     size([k IN keys(e) WHERE NOT k IN $meta AND
          any(kw IN $kws WHERE toLower(toString(e[k])) CONTAINS toLower(kw))]) AS prop_hits,
     size([x IN [y IN rels0 WHERE y.target IS NOT NULL] WHERE
          any(kw IN $kws WHERE toLower(x.target) CONTAINS toLower(kw))]) AS nbr_hits
   WITH e, rels, (label_hits*2 + prop_hits*2 + nbr_hits +
        CASE WHEN e.etype IN $intent THEN 5 ELSE 0 END) AS score
   WHERE label_hits>0 OR prop_hits>0 OR nbr_hits>0
   RETURN e.label, e.etype, properties(e), e.source_file, rels[..5], score
   ORDER BY score DESC LIMIT 15
   ```
   (이전엔 라벨 부분일치만 봤으나, 이제 값속성·관계까지 보고 의도 우선순위로 정렬한다.)
3. `_build_context`가 **`[구조화 사실]`(맨 앞) + `[문서 발췌]`(청크)** 순으로 컨텍스트 조립.
4. LLM에 "이 **자료만** 근거로 답하라"로 1회 호출 → 답변.
5. 전 과정을 `retrieval_detail.steps`로 기록 → 화면 "🔎 검색 과정"에 표시. 응답의 `facts`를 `FactsView`가 배지+칩으로
   그리고, "🕸 그래프보기"로 Cytoscape 관계 그래프 표시.

## 5. 이번 화면의 솔직한 해석 + 튜닝 포인트

답변의 핵심 수치(12~72개월, 엔카 24~60)는 사실 **청크**에서 나오고, 화면의 구조화 사실들(차량가격 산식 등)은
**보조 근거**로 붙었다. 두 가지를 시사한다:

1. 구조화 추출은 **작동**한다(`HandlingRule.rule`에 대출한도 산식이 정밀히 들어옴).
2. 현재 매칭이 **라벨 키워드 기반**이라, "할부/취급" 같은 일반어에 걸려 다소 주변적인 카드도 올라온다.
   정작 개월수·등급 값을 가진 핵심 카드가 항상 1순위는 아님.

**개선 방향 → 적용 상태 (2026-06-30)**
1. ✅ **구현됨** — 매칭을 라벨뿐 아니라 **값 속성·이웃(관계 대상) 라벨**까지 확대.
   `_structured_facts`가 `label_hits*2 + prop_hits*2 + nbr_hits` 로 점수화해 정렬.
2. ✅ **구현됨** — 질문 의도별 **etype 가산점**(`_intent_etypes`): 개월수·취급→`Product/RateGrade`,
   금리·등급·네고→`RateGrade/Nego`, 판정→`Decision`, 엔카·연식·차량→`VehicleCondition`.
   매칭 etype이 의도와 맞으면 +5점.
3. ☐ **운영** — `build_structured`를 `--limit` 없이 전체 적재하면 커버리지가 올라가 더 정밀한
   사실이 상위로 온다(사내망에서 실행).

> 갱신: `_structured_facts(question, kws)` 로 시그니처 변경(질문 의도 반영). 쿼리는 읽기전용 유지.
> UI "🔎 검색 과정" 단계 문구도 "라벨·값속성·관계 매칭 + 의도 우선 etype"으로 노출됨.

## 6. 컨텍스트 구성 원칙 — 검색 품질의 핵심 ★

정밀 검색만으로 정답이 나오지 않는다. 답변 정확도는 근거를 **무엇을·어떤 순서로·어떤 권위로**
컨텍스트에 담느냐에 좌우된다. 구조화 사실은 **고정밀일 때만 이득**이며, 저품질·부분적이면 청크의
뉘앙스를 가려 오답을 낸다(예: "프로모션 법인 가능?"의 `(개인고객 限)`은 청크에만 존재).

적용(코드 반영, 2026-06-30):
1. `_structured_facts` — 저점수 컷(`score ≥ 3`, 상위 8)으로 주변적 fact 제외.
2. `_build_context` — **문서 발췌(1차 근거)를 앞에**, 구조화 사실은 **보조로 뒤에**(컨텍스트 라벨에 "상충 시 발췌 우선" 명시).
3. `search_compare` — 구조화 사실을 양쪽에 동일 적용 → 비교 뷰 Hybrid = 기본 검색과 동일 파이프라인.

원칙: 1차 근거는 원문(청크), 그래프 사실은 정밀값·검증 보조 — 구조화는 정밀도를 *보강*하되 원문을 *대체하지 않는다*.

## 7. 발전 — 수치형은 "결정적 테이블"로 승격 (2026-07-02)

구조화 사실을 LLM 컨텍스트의 *보조*로 넣는 방식은 여전히 LLM 해석에 의존한다. 정밀값이 반드시 정확해야 하는
수치형 질의(금리등급·네고·프로모션)는 한 걸음 더 나아가, **의도 라우팅**으로 분기한 뒤 **큐레이션된 테이블(CSV)의
값을 템플릿으로 조립**해 LLM 없이 답한다(빈응답·환각 원천 제거).

- 라우팅·경로: `13_의도라우팅_개선계획.md`
- 규칙(정책·템플릿)·프롬프트: `14_답변규칙_카탈로그.md`, 코드 `app/search/{router,tables,answer_rules,prompts}.py`

정리하면 답변 근거는 3층이다: **① 결정적 테이블(정밀값, 최우선) → ② 원문 청크(1차 서술 근거) → ③ 구조화 사실(보조)**.

## 요약

"🧩 구조화 사실"은 **답변이 어떤 정제된 그래프 사실에 기반했는지 보여주는 설명가능성 장치**이자, 깨지기 쉬운
청크 검색을 정밀값으로 보강하는 레이어다. 지금은 라벨 매칭이라 보조적이지만, 매칭을 값·관계까지 넓히면
이 영역이 답변의 **1차 근거**로 올라설 수 있다.
