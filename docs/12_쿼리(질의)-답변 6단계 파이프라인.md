전체 쿼리 동작 시나리오


예시 질의

-  금리등급 2등급일 때 적용 될 수 있는 최저 금리가 궁금해

최종 답변

 - 금리등급 2등급시 21.0% G/L금리이며 내국인 네고 1%(N1) 적용됩니다.
   추가로 거점장 네고 11.0%, 증빙 네고 15.0%, HJ 네고 18.0%가 가능합니다.


쿼리(질의)에 답변할 때 6단계 파이프라인 실행.

1단계 — 질의 라우팅 

키워드 휴리스틱으로 분류합니다.
- 수치 힌트: "금리" ✓, "등급" ✓, 숫자 "2" ✓ → has_num = True
- 규칙 힌트("가능", "까지", "취급" 등): 질의에 없음 → has_rule = False
- 결과: route = "numeric" (수치형)

2단계 — 쿼리 확장 

동의어 사전에서 "금리" → "이자율", "rate", "G/L금리" 치환 변형 4개를 생성합니다. 다만 현재 코드에서 이 변형들은 검색에 실제 투입되지 않고 트레이스 표시용으로만 쓰입니다 (app/api.py:64-65).

3단계 — 구조화 테이블 조회  ← 답변이 여기서 결정됨

lookup()은 잔가군 → 조건 → 금리 → 범용조건 순으로 시도합니다. 

1. 정규식 (\d+)\s*등급이 "2등급"에 매치 + 질의에 "금리" 포함 → grade=2
2. "외국인" 없음 → customer_type="내국인", "N개월" 없음 → 최소 기간 구간 선택
3. SQL 조회 (실제 DB 확인 결과):
  - rate_grade 테이블: 내국인, 2등급, 12개월~ → G/L금리 21.0% (유효일자 2026-03-01, 출처 9_샘플_중고승용 상품운영기준.pdf p.1)
  - nego_rule 테이블 (grade_from<=2<=grade_to): internal → 1%(N1), 거점장 → 11.0%, 증빙 → 15.0%, HJ → 18.0%
4. 질의에 NICE 점수("NNN점")가 없어 NICE 분기는 건너뛰고, 파이썬 f-string 템플릿으로 문장 조립 (tables_lookup.py:63-75):

▎ "금리등급 2등급시 21.0% G/L금리" + "이며 내국인 네고 1%(N1) 적용됩니다" + ". 추가로 거점장 네고 11.0%, 증빙 네고 15.0%, HJ 네고 18.0%가 가능합니다."

받으신 답변과 글자 단위로 동일합니다. 모든 수치는 DB 행에서 그대로 온 것이고, LLM은 이 문장 생성에 전혀 관여하지 않습니다.

4단계 — 하이브리드 검색 (retriever)

테이블에서 정답이 나왔지만 /ask는 항상 with_trace=True이므로 근거 패널 표시용으로 검색을 수행합니다 :
- 벡터 검색: 상위 5개 청크
- 그래프 탐색: 히트된 문서 + 질의 토큰("금리등급" 등)과 라벨이 겹치는  노드를 시드로  1홉 확장 →  "근거 노드"(예: 금리등급 / G/L 금리 / 조정금리(NEGO) 노드)가 이것입니다

5단계 — 재랭킹 (reranker)

테이블 결과를 score 1.0으로 최상위에 고정하고, 벡터 청크를 점수순으로 뒤에 붙입니다.

6단계 — 답변 생성 (generator)




참고할 만한 점

질문이 "최저 금리"였는데, lookup_rate()는 최저금리를 계산하지 않고 "G/L금리 + 적용 가능한 네고 목록"을 나열하는 구조입니다. 
즉 "21.0%에서 네고를 빼면 최저 얼마"라는 산술은 의도적으로 수행하지 않고, 규정 수치를 그대로 제시해 상담사가 판단하게 합니다. 
또한 개월수를 지정하지 않으면 term_min_months 최소 구간(12개월) 행이 자동 선택된다는 점도 답변 해석 시 유의할 부분입니다.





라우터(router)는 LLM 없이 키워드 휴리스틱만으로 3가지 유형을 분류합니다. 기준은 두 개의 힌트 목록입니다.

판정 재료

① 수치 힌트 (NUMERIC_HINTS) — 질의에 하나라도 있으면 has_num = True

("금리", "등급", "개월", "한도", "금액", "점", "%", "네고", "nego", "이자율")
추가로 정규식 \d — 질의에 숫자가 하나라도 있어도 has_num = True입니다. 이 목록은 소문자 변환 후 비교하므로 "NEGO" 같은 대문자도 잡힙니다.

② 규칙 힌트 (RULE_HINTS) — 질의에 하나라도 있으면 has_rule = True

("가능", "까지", "판정", "슬라이딩", "대상", "여부", "취급")
"~가능해?", "몇 등급까지?", "R판정", "취급 조건" 같은 가부/범위/규칙 질문을 포착하는 단어들입니다. 이쪽은 원문 그대로 비교합니다.

분류 규칙 (router)

┌─────────┬──────────┬─────────────┬──────────────────────────────────────────────┐
│ has_num │ has_rule │    결과      │                     예시                      │
├─────────┼──────────┼─────────────┼──────────────────────────────────────────────┤
│ ✓       │ ✓        │ mixed       │ "듀얼상품 금리등급 몇 등급까지 취급 가능해?"           │
├─────────┼──────────┼─────────────┼──────────────────────────────────────────────┤
│ ✓       │ ✗        │ numeric     │ "금리등급 2등급 최저 금리"                         │
├─────────┼──────────┼─────────────┼──────────────────────────────────────────────┤
│ ✗       │ (무관)    │ descriptive │ "성능점검기록부가 뭐야?"                           │
└─────────┴──────────┴─────────────┴──────────────────────────────────────────────┘

지난 질의 "금리등급 2등급일 때 적용 될 수 있는 최저 금리가 궁금해"는 "금리"·"등급"·숫자 "2"로 has_num=True, 
규칙 힌트는 없어서("될 수 있는"은 "가능"과 다른 문자열이라 매치 안 됨) numeric이 된 것입니다.

라우팅 결과가 파이프라인에 미치는 영향 

테이블 조회가 실패했거나 route가 descriptive/mixed면 벡터·그래프 검색 결과를 답변 근거로 사용합니다. numeric이면서 테이블이 매칭되면 테이블 답이 그대로 최종 답변이 됩니다.

참고로 docstring(router.py:3-4)에 명시되어 있듯, 게이트웨이 연결 시 LLM 분류로 보강하는 것을 염두에 둔 설계지만 현재 구현은 휴리스틱만 사용합니다. 
즉 "될 수 있는"처럼 힌트 목록에 없는 표현은 놓치는 한계가 있고, 이는 동의어 사전(expander)이나 힌트 목록 확장으로 보완해야 하는 부분입니다.
