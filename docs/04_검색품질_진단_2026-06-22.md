# 검색 품질 진단 — 2026-06-22

## 결론
샘플 질문들의 정답값(12~72개월, 21.0% G/L, 듀얼 7등급, 엔카 슬라이딩 조건, R판정 취급불가)은
**현재 그래프(graph.json, 86노드)에 데이터로 존재하지 않는다.** 따라서 Opus·HCX 어느 모델로도
정확한 답을 낼 수 없다. → **모델 문제가 아니라 데이터(추출/적재) 커버리지 문제.**

## 근거
- 노드는 개념(label)만 보유. 속성 키 = `label / norm_label / file_type / source_file / community`
  뿐이며 **값(수치·기준·판정결과) 필드가 없음.**
- 핵심 노드 이웃 점검:
  - `대출 취급 가능 개월수` → `has_property → 중고승용 상품운영기준` (값 "12~72개월" 없음)
  - `엔카 슬라이딩` → `has_property → 엔카 무수수료 상품` (조건 없음)
  - `금리등급` → `has_property → G/L 금리` (등급별 금리표 없음)
  - `R판정/B판정` → `governs → 심사기준(CSS)` ("필터링 취급 불가" 결과 없음)
- 반면 원본 문서(FAQ xlsx 52행 등)에는 구체값이 행 단위로 존재.
  예) "한도 7천만원 / 업력 9개월 / 설정율 60%" 등.

## 권고 (품질 개선 경로)
1. **FAQ(xlsx) 결정적 적재** — 가장 빠른 효과. 각 FAQ 행(질문/카테고리/답변)을 노드로 적재하고
   답변 텍스트를 속성으로 보유 → FAQ형 질문에 즉시 정답 반환.
2. **운영기준 문서 적재** — 금리표·취급기간·판정기준 등 표/수치를 노드 속성 또는 값 노드로 추출
   (xlsx/표는 결정적 매핑, 서술형은 LLM 추출). `ingestion/parsers.py` + `extract.py` 구현.
3. **하이브리드(텍스트 청크 연결)** — 각 엔티티에 source 청크 텍스트를 붙여, 그래프로 관련 청크를
   찾고 그 원문으로 답을 생성(현재는 source_file만 있고 본문 텍스트가 없음).
4. 위 보강 후 `eval_queries.py`로 Opus↔HCX 정확도 재비교.
