## 오늘 한 일 (요약)

### 1) Hybrid 검색 라이브 안정화
- **인덱스 이름 자동 탐지** — `hybrid.py`가 `:Chunk(embedding)`/`:Chunk(text)`의 ONLINE 인덱스를
  이름과 무관하게 탐지(`chunkVec` vs `chunk_embedding_index` 환경차 대응).
- **생성 후 ONLINE 대기** — `setup_hybrid`가 `CALL db.awaitIndexes()`로 대기 후 검증(POPULATING 오판 중단 버그 수정).
- **차원 충돌 해결** — 공유 DB의 타 프로젝트 벡터 인덱스(3072차원)와 1024 충돌 발견 → DB 초기화로 격리.
- 결과: 청크 189개 bge-m3 임베딩 적재, 두 인덱스 ONLINE, Hybrid 검색 정상 동작 확인.

### 2) DB 초기화/격리 + 진단 스크립트
- `app/scripts/reset_db.py` — 노드/관계 + 인덱스/제약 전체 삭제(`--yes` 필수). 공유 영화 데이터·충돌 인덱스 제거.
- `app/scripts/check_hybrid.py` — 임베딩·인덱스·벡터/풀텍스트 단계별 진단(폴백 원인 추적).

### 3) 구조화 그래프를 검색에 반영
- `cypher_qa._structured_facts(question, kws)` — `origin='structured'` 엔티티를 조회 →
  `[구조화 사실]` 블록으로 LLM 컨텍스트에 주입(깨진 표 대신 정밀값). 답변 응답에 `facts` 포함.
- **매칭 고도화(개선 ①②)**: 라벨뿐 아니라 **값 속성·관계 대상**까지 매칭해 점수순 정렬
  (`label_hits*2 + prop_hits*2 + nbr_hits`), `_intent_etypes`로 질문 의도에 맞는 etype에 **+5점**.
  기본 모델은 이후 **HCX-30B-Text**로 변경(배치 추출은 Qwen 권장).

### 4) 벡터 vs Hybrid 비교
- `hybrid.vector_only_retrieve()`, `cypher_qa.search_compare()`, **`POST /search/compare`** 추가.

### 5) 검색 과정 상세 노출
- `retrieval_detail.steps`에 전 과정(임베딩→벡터→풀텍스트→RRF→구조화사실→컨텍스트→LLM) 단계 기록.

### 6) 새 LLM 모델 — HCX-30B-Text
- `registry.py` `hcx30` 추가(`hcx-agent-05`, thinking). 사내 게이트웨이 `:11000/v1/chat/completions`.
- `ModelSpec.extra_body` + factory가 OpenAI SDK `extra_body`로 `chat_template_kwargs.thinking` 전달
  (초기 `model_kwargs` 방식은 `chat_template_kwargs` 미허용 TypeError → `extra_body`로 수정해 연결 성공).
- `.env` `HCX30_*` 추가. 기본 모델을 Qwen3.6 → **HCX-30B-Text**로 확정(대화·답변용). 배치 추출(build_structured)은 비추론 Qwen 권장.
