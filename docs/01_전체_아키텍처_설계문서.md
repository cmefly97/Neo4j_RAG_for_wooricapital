# Graphify형 지식그래프 검색 시스템 — 상세 설계 문서

> 문서/지식베이스 기반 GraphRAG 검색 플랫폼 · UI 중심 설계
> 작성일: 2026-06-22

---

## 1. 개요

### 1.1 목적

비정형 문서(사내 위키, PDF, Notion, 보고서 등)를 **지식그래프(Knowledge Graph)** 로 변환하고, 사용자가 **UI 기반으로 자연어 검색 + 그래프 탐색**을 수행할 수 있는 검색 시스템을 구축한다. 단순 키워드/벡터 검색의 한계(맥락 단절, 멀티홉 추론 불가)를 그래프 구조로 보완하는 **GraphRAG** 방식을 채택한다.

### 1.2 핵심 가치

- **멀티홉 추론**: "A 정책과 연결된 부서의 담당자가 작성한 문서"처럼 여러 단계를 거치는 질의에 답한다.
- **출처 추적성(Traceability)**: 모든 답변 노드가 원본 문서 청크로 역추적된다.
- **시각적 탐색**: 답을 텍스트로만 받는 게 아니라, 엔티티 간 관계를 그래프로 보며 탐색한다.
- **하이브리드 검색**: 의미 기반(벡터) + 구조 기반(그래프 순회)을 결합해 정확도를 높인다.

### 1.3 설계 결정 요약

| 항목 | 선택 | 근거 |
|---|---|---|
| 데이터 소스 | 비정형 문서/지식베이스 | 요구사항 |
| 검색 패러다임 | GraphRAG (그래프 + 벡터 + LLM) | 멀티홉·맥락 보존 |
| 저장소 | 그래프 DB (Neo4j) + 벡터 인덱스 | Cypher + 네이티브 벡터 동시 지원 |
| UI | 그래프 시각화 / 자연어 검색창 / 패싯 필터 / 상세 패널 | 요구사항 |

## 2. 전체 아키텍처

### 2.1 4계층 구조

```
┌─────────────────────────────────────────────────────────────┐
│  ① Ingestion & KG Construction (오프라인/배치)               │
│  문서 수집 → 청킹 → 엔티티·관계 추출(LLM) → 정규화 →         │
│  그래프 적재 + 임베딩 생성                                    │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│  ② Storage Layer                                             │
│  Graph DB (Neo4j)  |  Vector Index  |  Object/Doc Store      │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│  ③ Retrieval & Reasoning (온라인)                            │
│  쿼리 라우터 → 하이브리드 검색(벡터+그래프) → 서브그래프 추출 │
│  → 컨텍스트 조립 → LLM 답변 생성                              │
└─────────────────────────────────────────────────────────────┘
                              │
                              ▼
┌─────────────────────────────────────────────────────────────┐
│  ④ Application & UI                                          │
│  API Gateway → 자연어 검색창 / 그래프 뷰 / 패싯 / 상세 패널  │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 데이터 흐름 (Query Path)

1. 사용자가 자연어 질의 입력
2. **쿼리 라우터**가 질의 유형 분류 (사실 질의 / 관계 탐색 / 요약형)
3. **벡터 검색**으로 의미적으로 가까운 엔티티·청크 시드(seed) 확보
4. 시드 노드 기준 **그래프 순회**(1~2 홉)로 연결된 서브그래프 추출
5. 청크 텍스트 + 그래프 관계를 **컨텍스트로 조립**
6. **LLM**이 출처가 명시된 답변 생성 + UI에 서브그래프 반환

## 3. ① 데이터 수집 & 지식그래프 구축 파이프라인

### 3.1 단계별 처리

**(1) 문서 수집 (Ingestion)**
- 대상 형식(확정): **xlsx, docx, pdf, md** 4종
- 커넥터: 파일 업로드, (선택) Google Drive/공유 폴더 연동
- 메타데이터 보존: 파일명, 형식, 제목, 작성자, 작성일, 출처 경로, 권한 정보
- 형식별 파이프라인을 분기해 처리한다 (아래 3.1-(2) 참조)

**(2) 형식별 파싱 & 청킹 (Chunking)**

문서를 크게 **비정형(서술형) 텍스트**(docx/pdf/md)와 **정형(표) 데이터**(xlsx, 일부 pdf/docx 내 표)로 나눠 다르게 처리하는 것이 핵심이다.

| 형식 | 파서 | 처리 전략 |
|---|---|---|
| **md** | 마크다운 파서 | 헤더(`#`) 기준 계층 청킹. 구조가 이미 명확해 손실 적음. 코드블록·표 보존. |
| **docx** | `python-docx`, Unstructured | 제목/스타일(Heading) 기반 청킹. 본문 텍스트는 LLM 추출, 내부 **표는 정형 경로**로 분리. |
| **pdf** | LlamaParse / Unstructured / PyMuPDF | 레이아웃 인식 추출. 스캔본은 **OCR** 적용. 표 감지 시 정형 경로로. 페이지·섹션 메타 보존. |
| **xlsx** | `openpyxl` / `pandas` | **표→그래프 직접 매핑**(LLM 추출 최소화). 시트=엔티티 집합, 헤더 행=속성, 데이터 행=엔티티 인스턴스. |

- 비정형 텍스트 청킹: 의미 단위(semantic) 또는 헤더 기반, 300~800 토큰 / 오버랩 10~15%
- 모든 청크에 `source_file`, `sheet/page/heading`, `chunk_id` 메타 부착 → 출처 추적 보장

**(2-1) xlsx(표 데이터) 전용 처리**
- 표는 이미 구조화돼 있으므로 LLM 자유 추출보다 **결정적 매핑(deterministic mapping)** 이 정확하고 저렴하다.
- 매핑 규칙 예시
  - 시트/테이블 → 엔티티 타입 (예: `상품목록` 시트 → `Product` 노드)
  - 헤더 컬럼 → 노드 속성 또는 관계 (예: `담당부서` 컬럼 → `BELONGS_TO` 관계)
  - 데이터 행 → 엔티티 인스턴스 1개 (행의 고유 키로 노드 ID 생성)
  - 외래키성 컬럼(다른 시트 참조) → 엔티티 간 엣지로 연결
- 병합 셀·다중 헤더·비정규 레이아웃은 전처리(헤더 평탄화)로 정규화
- 규칙으로 처리 불가한 자유서술 셀(비고/설명 등)만 LLM 추출 보조 사용
- 장점: 표 기반 사실(수치·소속·코드 등)의 정확도와 재현율이 텍스트 추출보다 높음

**(3) 엔티티·관계 추출 (LLM Extraction)** — 가장 중요한 단계
- **스키마 가이드 추출**: 허용 엔티티 타입/관계 타입을 사전에 정의해 LLM에 제약을 건다 (스키마 드리프트 방지).
- 도구: LangChain `LLMGraphTransformer`, LlamaIndex `PropertyGraphIndex`, 또는 Neo4j `LLM Knowledge Graph Builder`
- Tool/structured-output 모드를 써서 `(노드, 관계, 속성)` JSON으로 강제 출력
- 각 추출 결과에 **원본 청크 ID를 첨부**해 출처 추적 보장

**(4) 엔티티 정규화 & 해소 (Entity Resolution)**
- 동일 개체 병합 (예: "우리캐피탈" = "Woori Capital")
- 기법: 임베딩 유사도 + 문자열 유사도 + 규칙. 임계값 초과 시 후보 병합
- ⚠️ 스케일이 커질수록 핵심 난제. 사람이 검수하는 reconciliation 큐를 둔다.

**(5) 적재 & 임베딩**
- 노드/관계 → Neo4j 적재 (`MERGE`로 중복 방지)
- 각 청크·엔티티 임베딩 생성 → 벡터 인덱스 저장
- (선택) **커뮤니티 탐지**: Leiden 알고리즘으로 클러스터링 후 커뮤니티 요약 생성 → 광역(global) 질의 대응

### 3.2 온톨로지/스키마 설계

도메인 특화 온톨로지를 먼저 정의하면 추출 품질이 크게 오른다. 예시(금융/사내 문서 도메인):

```
엔티티(Node) 타입:
  - Document(문서), Person(인물), Organization(부서/조직),
    Policy(정책/규정), Product(상품), Concept(개념), Event(이벤트)

관계(Edge) 타입:
  - AUTHORED_BY (문서 → 인물)
  - BELONGS_TO (인물 → 조직)
  - REFERENCES (문서 → 정책)
  - RELATED_TO (개념 ↔ 개념)
  - MENTIONS (문서 → 엔티티)
  - DERIVED_FROM (청크 → 문서)   ← 출처 추적의 핵심
```

> 실무 팁: 처음부터 전체 코퍼스를 넣지 말고, 특정 도메인 **50~100개 문서**로 스키마와 추출 품질을 검증한 뒤 확장한다.

## 4. ② 저장소 계층

| 저장소 | 역할 | 후보 기술 |
|---|---|---|
| **그래프 DB** | 엔티티·관계 저장, Cypher 순회 | **Neo4j**(권장), Memgraph, Amazon Neptune |
| **벡터 인덱스** | 의미 검색용 임베딩 | Neo4j 네이티브 벡터 인덱스, 또는 Qdrant/pgvector/Weaviate |
| **문서/객체 저장소** | 원본 파일·청크 원문 | S3/MinIO + RDB(메타데이터) |
| **캐시** | 쿼리/임베딩 캐시 | Redis |

**Neo4j를 핵심으로 추천하는 이유**: 속성 그래프(Cypher)와 벡터 인덱스를 한 엔진에서 모두 지원해 하이브리드 검색 구현이 단순해진다. 별도 벡터 DB 운영 부담을 줄일 수 있다. 단, 초대형(수억 노드) 또는 독립 확장이 필요하면 벡터 DB를 분리한다.

## 5. ③ 검색 & 추론 계층 (GraphRAG)

### 5.1 쿼리 라우터

질의 유형에 따라 검색 전략을 분기한다.

| 질의 유형 | 예시 | 전략 |
|---|---|---|
| **로컬(Local)** | "X 정책의 담당 부서는?" | 엔티티 시드 → 근방 서브그래프 |
| **글로벌(Global)** | "올해 리스크 정책 변화 요약" | 커뮤니티 요약 집계 |
| **멀티홉** | "A와 연결된 B가 작성한 문서" | 벡터 시드 + 다중 홉 순회 |
| **순수 의미** | "비슷한 내용의 문서 찾아줘" | 벡터 검색 위주 |

### 5.2 하이브리드 검색 알고리즘

```
1. 벡터 검색: 질의 임베딩 → top-k 엔티티/청크 시드 확보
2. 그래프 확장: 시드 노드에서 N홉 이웃 + 관계 수집 (Cypher)
3. 랭킹: (벡터 유사도 + 그래프 중심성 + 경로 길이) 가중 결합
4. 컨텍스트 조립: 선택된 청크 원문 + 관계 트리플을 LLM 프롬프트로 구성
5. 답변 생성: LLM이 인용(citation) 포함 답변 + 반환 서브그래프 산출
```

### 5.3 LLM 연동

- **Query → Cypher 변환**: 구조가 명확한 질의는 LLM이 직접 Cypher 생성(LangChain `GraphCypherQAChain`). 잘못된 쿼리 방지를 위해 스키마 주입 + 검증 단계 필수.
- **답변 생성**: 조립된 컨텍스트 기반 RAG. 환각 억제를 위해 "컨텍스트에 없으면 모른다고 답하라" 가드레일.
- **모델 선택**: 추출용(정확도 우선)과 답변용(속도/비용 균형)을 분리. 한국어 비중이 높으면 한국어 성능 검증된 모델 선택.

### 5.4 품질·한계 대응

- 연구상 답변 엔티티의 약 65%만 그래프에 존재 → **누락 엔티티가 곧 추론 단절**. 추출 재현율 모니터링과 벡터 폴백을 병행한다.
- 평가 지표: 검색 정확도(Recall@k), 답변 충실도(faithfulness), 출처 정확도, 응답 지연(latency).

### 5.5 컨텍스트 구성 원칙 — 검색 품질의 핵심 ★

> **정밀 검색 ≠ 정답. 정답은 컨텍스트 구성에서 완성된다.** LLM 답변 정확도는 검색된 근거를
> **무엇을 · 어떤 순서로 · 어떤 권위로** 컨텍스트에 담느냐에 크게 좌우된다. 좋은 근거를 찾아도
> 구성이 나쁘면 오답이 난다.

- **구조화 사실은 고정밀일 때만 이득** — 저품질·부분적(단서 누락)·주변적 매칭이면 오히려 청크 원문의
  뉘앙스를 가려 오답을 유발한다. (예: "프로모션 법인 가능?"의 정답 조건 `(개인고객 限)`은 청크 원문에만 존재)
- **오답 유발 3요인** — ① 앞자리(primacy) 배치 ② "사실"로서의 권위 부여 ③ 추출 손실 압축·매칭 노이즈.
- **적용(2026-06-30)**
  1. **저점수 구조화 사실 컷** (`score ≥ 3`, 상위 8) — 주변적 매칭 제외.
  2. **문서 발췌를 1차 근거로 앞에**, 구조화 사실은 **보조로 뒤에** + 프롬프트 "상충 시 발췌 우선".
  3. **비교 뷰 Hybrid = 기본 검색과 동일 파이프라인** (구조화 사실은 검색방식과 무관하게 동일 적용).
- **원칙** — 1차 근거는 **원문(청크)**, 그래프 사실은 **정밀값·검증 보조**. 구조화 레이어는 정밀도를 *보강*하되 원문을 *대체하지 않는다*.

## 6. ④ 애플리케이션 & UI

### 6.1 화면 레이아웃

```
┌──────────────────────────────────────────────────────────┐

│               │                          │               │
│  □ 엔티티 타입 │      ○──○                │  [선택 노드]  │

```

### 6.2 핵심 기능 명세

**(1) 자연어 검색창**
- 질의 입력 → AI 답변(출처 인용) + 관련 서브그래프 자동 렌더
- 자동완성: 엔티티명 제안, 최근 검색 기록
- 답변 내 인용 클릭 → 해당 노드/원문 하이라이트

**(2) 그래프 시각화 캔버스**
- 노드(엔티티)·엣지(관계) 인터랙티브 렌더
- 인터랙션: 노드 클릭(상세), 더블클릭(이웃 확장), 드래그, 줌/팬
- 시각 인코딩: 타입별 색상·아이콘, 중요도(중심성)별 노드 크기
- 레이아웃: force-directed 기본, 계층형/방사형 토글
- "확장(expand)" 버튼으로 점진적 탐색 (전체 그래프를 한 번에 그리지 않음)

**(3) 패싯/필터 탐색**
- 엔티티 타입, 관계 타입, 작성자, 기간, 출처별 필터
- 다중 선택 + 실시간 그래프 갱신
- 활성 필터 칩(chip) 표시 및 개별 해제

**(4) 결과 상세 패널**
- 선택 노드의 속성, 출처 문서, 연결 관계 목록
- 원문 발췌(청크) 표시 → 원본 문서로 딥링크
- "이 노드 중심으로 다시 탐색" 액션

### 6.3 UI 기술 스택

| 영역 | 권장 기술 | 비고 |
|---|---|---|
| 프레임워크 | React + TypeScript | 컴포넌트·상태관리 성숙도 |
| 그래프 렌더 | **Cytoscape.js** (알고리즘·레이아웃 풍부) 또는 **Sigma.js + graphology** (10만+ 노드, WebGL) | 규모에 따라 선택 |
| 대안 | react-force-graph (3D 필요 시), vis-network (즉시 인터랙션) | |
| 상태관리 | Zustand 또는 React Query(서버 상태) | |
| 스타일 | Tailwind CSS + 컴포넌트 라이브러리 | |
| 시각화 보조 | D3 (커스텀 차트/패싯 통계) | |

**그래프 라이브러리 선택 가이드**
- 노드 수 수천 이하 + 풍부한 분석/레이아웃 필요 → **Cytoscape.js**
- 노드 수 수만~10만+ + 성능 우선 → **Sigma.js + graphology** (React 공식 바인딩 `@react-sigma`)
- 3D 탐색 경험 → react-force-graph

### 6.4 백엔드/API

- API 게이트웨이: FastAPI(Python) — LLM/그래프 생태계와 친화적
- 엔드포인트 예시
  - `POST /search` — 자연어 질의 → 답변 + 서브그래프
  - `GET /graph/expand?node_id=` — 노드 이웃 확장
  - `GET /node/{id}` — 노드 상세 + 출처
  - `GET /facets` — 필터 옵션 메타데이터
- 인증/권한: 문서 권한(ACL)을 그래프 노드 메타에 반영해 **검색 결과 권한 필터링**
- 스트리밍: LLM 답변 SSE 스트리밍으로 체감 지연 개선

## 8. 단계별 구축 로드맵

**Phase 1 — PoC (3~4주)**
- 50~100개 문서로 온톨로지·추출 파이프라인 검증
- Neo4j 적재 + 단순 벡터 검색
- 최소 UI: 검색창 + 그래프 뷰

**Phase 2 — 하이브리드 검색 (4~6주)**
- 벡터+그래프 하이브리드 리트리버, 쿼리 라우터
- 패싯 필터, 상세 패널, 출처 추적
- 엔티티 해소 파이프라인

**Phase 3 — 고도화 (6주+)**
- 커뮤니티 요약(글로벌 질의), 평가/모니터링 체계
- 권한 기반 검색, 증분 업데이트(문서 변경 반영)
- 성능 최적화(대규모 그래프 WebGL 렌더, 캐싱)
