# 질의 의도 라우팅 개선계획 (2026-07-02)

> 근거 문서: `docs/12_쿼리(질의)-답변 6단계 파이프라인.md`(참고 아키텍처)
> 대상 코드: `backend/app/search/cypher_qa.py`, `hybrid.py`
> 목표: **질문 의도를 파악해 경로를 분기(라우팅)** — 수치형은 결정적 답(LLM 무개입), 서술형은 Hybrid+LLM.

## 1. 문제 인식 (현재 파이프라인의 한계)

현재 `search()`는 의도와 무관하게 **모든 질문이 동일 경로**(Hybrid 검색 → 구조화 사실 보조 → LLM 1회)를 탄다.
`_intent_etypes`는 구조화 사실의 etype 가산점만 조정할 뿐, **경로를 바꾸지 않는다.**

그 결과:
- 수치형 질의("금리등급 2등급 최저금리", "프로모션 기준")도 LLM이 깨진 PDF 표 청크를 해석 →
  **빈 응답·환각·부정확**이 반복(로그: `LLM_ANSWER chars=0`, 오답 관측).
- 정답 값이 원문 표에 있어도 텍스트화 과정에서 뭉개져 **LLM이 못 읽는다.**

## 2. doc-12에서 채택하는 핵심 아이디어

1. **라우터**(키워드 휴리스틱): `numeric` / `mixed` / `descriptive` 3분류.
   - `NUMERIC_HINTS`(금리·등급·개월·한도·금액·점·%·네고·이자율 + 숫자)
   - `RULE_HINTS`(가능·까지·판정·슬라이딩·대상·여부·취급)
2. **수치형 결정적 답변**: 정제 테이블(rate_grade/nego_rule/promotion)에서 값을 뽑아
   **템플릿으로 문장 조립, LLM 무개입** → 빈응답·환각 원천 제거.
3. **재랭킹**: 테이블 결정 답을 최상위(score 1.0)로 고정, 청크는 근거 패널용.
4. **도메인 답변 규칙**: "최저금리"라도 `G/L − 네고` 산술을 하지 않고 **G/L + 적용 가능 네고 목록**을
   제시(상담사 판단). 개월수 미지정 시 **최소 구간** 자동 선택.

## 3. 우리 코드와의 갭

| doc-12 요소 | 현재(agent2) | 조치 |
|---|---|---|
| 라우터 numeric/mixed/descriptive | `_intent_etypes`(가산점만) | **router.py 신설** |
| 결정적 테이블 조회 → 템플릿 답변(LLM 무개입) | 없음(항상 LLM) | **tables.py + data/*.csv 신설** |
| 재랭크 테이블 score 1.0 핀 | RRF+벡터보장까지 | 결정적 답이면 short-circuit |
| rate_grade/nego_rule 정밀 테이블 | LLM 추출 :Entity(불안정) | **CSV 큐레이션(결정적)** |

## 4. 실행 계획

### 선결사항 — 결정적 테이블 기반 (LLM/Neo4j 무의존) ✅ 이번 작업
- `backend/data/rate_grade.csv`, `nego_rule.csv`, `promotion.csv` — 스키마 + 근거 있는 시드행 + TODO.
- `backend/app/search/tables.py` — CSV 로드, `lookup_rate/lookup_nego/lookup_promotion`,
  `answer_numeric(question)`(정규식으로 등급/프로모션 감지 → 템플릿 답 조립).
- **원천 데이터는 큐레이션**: 깨진 PDF 표 대신 CSV에 실제 규정 값을 채운다(사용자 검수).

### Phase 1 — 라우터 + 파이프라인 통합 ✅ 이번 작업
- `backend/app/search/router.py` — `route_intent(question)` → `{route, has_num, has_rule, hits}`.
- `cypher_qa.search()` 통합:
  1. `route = route_intent(question)` → 로그 `ROUTE=<route>`.
  2. **결정적 short-circuit**: `tables.answer_numeric()`가 답을 만들면 그대로 반환(LLM 미호출, `status=ok_table`).
  3. 아니면 기존 Hybrid+LLM 경로. route는 구조화 사실 우선순위/로깅에 활용.

### Phase 2 — 신뢰도·확장 (다음)
- 가드레일: 근거 점수 낮으면 "규정 범위 밖" 안내(환각 차단).
- 재랭킹: mixed에서 테이블 사실 상위 고정 + 청크 근거.
- 답변 검증(faithfulness): 답변 수치가 근거에 실제 존재하는지 대조.

### Phase 3 — 지능화
- 애매 표현("될 수 있는")까지 잡도록 힌트 확장 + 게이트웨이 시 **LLM 의도 분류** 보강.
- 비교/멀티홉 다중검색, 평가셋으로 라우팅 정확도 측정.

## 5. 기대효과
- 수치형: 빈응답·환각 **제거**, 정답 값 **결정적** 제공, LLM 호출 **절감**(지연·비용↓).
- 서술형: 기존 Hybrid+LLM 유지(품질 동일).
- 설명가능성: `ROUTE`·테이블 출처 로그로 어느 경로로 답했는지 추적.

## 6. 리스크 / 유의
- 결정적 답의 정확도는 **CSV 값의 정확도**에 달림 → 규정 원문으로 검수 필수.
- 시드행은 doc-12 예시·사용자 제시 모범답안 기반의 **참고값**이며, 전체 커버리지는 사용자가 채운다.
- 기존 폴백(빈응답 재시도, no_results, 읽기전용 가드)은 그대로 유지.
